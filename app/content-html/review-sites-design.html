Core feature requirements for this project were the ability to upload review content, and customize what metadata is shown on each page. We also needed to balance flexibility, since every package has its own unique needs, and uniformity, since reviewers might use dozens of different review sites over the course of a year and need to have a similar experience.

Since I didn’t have much time to complete this project, design consisted of quick, lo-fi wireframes, which I spent a few days workshopping with project stakeholders to nail down the most important aspects of layout and functionality. Some of the bigger design challenges, such as how to display complex data sets, didn’t emerge until I was in the middle of implementation, so we worked them out over the course of a few iterations on the production code. The screenshot below shows how we used color coded data, multiple tables, and additional column labels to squeeze response data into the required tabular format.

Unsurprisingly, we ran into some usability problems with the initial version of the review sites, so I added a few features to help reviewers navigate and make sense of the packages. A couple of examples are below—

  • The “guided tour” is a simple carousel with annotated screenshots introducing new reviewers to important features.
  • Package assemblers can annotate any metadata field with on-demand help text. Help links are indicated with dotted green underline.

Review sites have complex configuration options, so having to key all the configuration data into a spreadsheet every time you want to make a new site isn’t a great solution, usability-wise, for the folks putting these sites together.

When it became clear that we might be using the review sites for the long haul, I went back and built a GUI to help package assemblers configure packages without the need for painstakingly editing config spreadsheets.
