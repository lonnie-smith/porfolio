The initial application required site assemblers to enter configuration data into spreadsheets. These spreadsheets (in CSV format) were served alongside the app, which had to parse them at run time. I couldn’t find an off-the-shelf in-browser CSV parser, so I wrote a state machine parser as an Angular service.

The help feature is implemented in a very simple way. It needed to be easy for package configurers to add, remove, or change help text without touching application code. So they type the help information they want into a 3-column spreadsheet. The Angular app loads the help data, and then templates check for the presence of help content on every field and generate a help dialog if one is needed.

When it became apparent that we were likely to be using the review site stopgap for the long term, I built a GUI for configuring packages so that package creators wouldn’t have to laboriously hand-craft spreadsheets. Here’s a bit of the code for that package builder app.

  • The right way to write this application would be to store package configuration data in a database. That wasn’t an option, so instead I created an object model that could be serialized to JSON strings and stored in the browser’s memory (indexedDB). This provided a feasible way to persist data across multiple work sessions as a package creator assembled the review site.
  • I wrote an abstract Serializeable class that knows how to convert in-memory objects to and from JSON strings. I wrapped this up as an Angular factory so that it could be easily accessed throughout the application and so that it could be isolated for unit testing.
  • Most of the configuration objects used by the package builder app extend the Serializeable class with whatever custom properties or methods they need.
