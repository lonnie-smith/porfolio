Given the diversity of content layout—no two passages are alike—we knew we couldn’t reasonably build a CMS-like authoring system for passages. In order to get letter-perfect passage layouts, we hired designers and web developers to lay out each passage by hand.

My job was writing the shared code that would provide each passage with a shared set of interactive features like page navigation, dialog boxes, video playback, and user activity tracking. I implemented each feature in a separate AMD-style module. Passage authors mark up each passage with special class names to identify key features—say, the pages in a magazine article or the content that should appear in a dialog box. The correct modules are loaded based on the presence of elements in the DOM with these class names, and each module uses jQuery to manipulate the DOM and append user controls. Interoperation with the host application is handled through a JavaScript API I specced out.

The code samples below illustrate how I implemented the “Lookback Button” feature. For context, here’s a video of the feature in action.

First, we needed a succint way for passage authors to mark the beginning and end of a Lookback section. These sections can overlap and often span chunks of passage markup awkwardly (for example, a section could begin in the middle of one <p> element and end in the middle of another). Adding markup for lookback sections in and among the normal layout markup would be pretty cumbersome, so we just asked page authors to place a special element at the beginning, and another at the end, of lookback sections. Code sample 1 shows a snippet of a passage with <focus-start> and <focus-end> tags added. Code sample 2 shows the the function that litters the otherwise-readable page with the markup needed to highlight the relevant section.

Next, different passages use different navigation controls. For example, some passages use pages, while others scroll or are navigated by means of menus. Each navigation module therefore needs to handle moving to the lookback section in a different way. Each navigation control implements its own event handler, so each control can respond independently of the others. Code sample 3 shows the event handler used by the paging module.
